# Basta · Quality Assurance Report
**Data:** 7 de maio de 2026
**Escopo:** 24 HTMLs ativos (`01–24`) + foundation/components/partials
**Tester:** auditoria estrutural (estilo Google QA · web-app review)
**Versão revisada:** snapshot pré-produção

---

## Resumo executivo

O sistema é **estruturalmente sólido** e mostra alto rigor: todos os 24 HTMLs ativos têm `<html lang="pt-BR">`, `<title>`, viewport correto, foundation completo (5/5 imports), `100vh` sempre com fallback `100dvh`, sidebar/topbar consistentes entre páginas administrativas. Não há link interno quebrado.

Os achados estão concentrados em **três frentes**:

1. **Mobile UX nas páginas pré-login (01–11)** — 11 páginas com `:hover` ativo sem `@media (hover: none)`, causando "hover preso" em touch.
2. **Segurança em formulários** — campos `email` sem `autocomplete`, `password` sem `autocomplete=current-password/new-password` em 3 páginas críticas, 1 link `target="_blank"` sem `rel="noopener"`.
3. **Higiene de código** — `alert()` debug em 3 páginas, scripts inline volumosos (até 273 linhas), cópia inline do logo SVG em todos os arquivos (oportunidade de externalizar).

Severidades distribuídas: **2 Críticos**, **6 Altos**, **9 Médios**, **5 Baixos**.

Nenhum bloqueador de produção. Recomendação: corrigir Críticos + Altos antes de soft-launch público; Médios podem ser endereçados em sprint subsequente.

## 1. Consonância · estrutura e foundation

### ✓ Pontos fortes

- **100% dos arquivos** com `<html lang="pt-BR">`, `<meta viewport>`, `<title>`, `<meta theme-color>`.
- **5/5 imports do foundation** (tokens, reset, grid, typography, layout) em todos os 24 HTMLs.
- Sidebar admin: itens consistentes entre as 6 páginas admin (18–23) com 11 links únicos cada, mesmos `href`.
- Sidebar cliente: itens consistentes entre as 6 páginas cliente (12–17) com 6 links únicos cada.
- Logo SVG inline está presente uma única vez por arquivo (sem duplicação interna).

### ⚠ Achados

**[ALTO] CO-01 · Cópia inline do logo SVG em 24 arquivos**
- **O quê:** cada HTML traz `<symbol id="logo-basta">` inline, mesmo havendo `_partials/logo-symbol.html` disponível.
- **Impacto:** payload extra ~2,5 KB por página × 24 páginas = ~60 KB redundantes; manutenção duplicada (qualquer ajuste no path do logo precisa em 24 lugares).
- **Recomendação:** após avaliar a engine de templating do projeto, externalizar via `<use href="/_partials/logo-symbol.html#logo-basta">` ou via build-time include.

**[MÉDIO] CO-02 · 13.467 linhas de CSS in-page**
- **O quê:** estilos por página estão dentro de `<style>` inline. As páginas admin têm 600–1.000 linhas cada.
- **Impacto:** tudo que é específico de página fica acoplado a HTML — não cacheia entre rotas, não compartilha entre páginas similares.
- **Recomendação:** identificar padrões repetidos (kanban, list-view, finding cards, lead cards) e extrair para `_components/`. Ex.: `_components/kanban.css`, `_components/lead-card.css`, `_components/finding-card.css`.

**[BAIXO] CO-03 · `01_basta-landing.html` sem `<meta name="robots" noindex>`**
- **O quê:** todas as outras 23 páginas têm `noindex,nofollow`. A landing não tem.
- **Impacto:** se o sistema é pré-produção (snapshot), a landing pode acabar indexada acidentalmente.
- **Recomendação:** se o snapshot for público, adicionar `<meta name="robots" content="noindex,nofollow">` enquanto não for lançamento oficial. Se for produção, decisão de SEO.

## 2. Responsividade & Mobile UX

### ✓ Pontos fortes

- **Topbar mobile com `safe-area-inset-top`** em 11/12 páginas admin/cliente — respeita o notch do iPhone.
- **`100vh` sempre acompanhado de `100dvh` fallback** em 22/24 arquivos — layouts não quebram com a barra de URL móvel surgindo/sumindo.
- Breakpoints coerentes: 640/720/768/960/1024/1280px nas páginas mais complexas.
- CPF nos formulários consistentemente com `inputmode="numeric"` + `autocomplete="off"`.

### ⚠ Achados

**[CRÍTICO] MB-01 · 11 páginas com `:hover` ativo sem media query `(hover: none)`**

Páginas afetadas: `01, 02, 03, 04, 05, 06, 07, 08, 09, 10, 11` (todas as públicas e pré-login).

- **O quê:** estados hover ativos podem "grudar" em telas touch após um tap, deixando botões/cards visualmente alterados até o usuário tocar em outro lugar.
- **Impacto:** experiência confusa em mobile no caminho mais crítico do funil — o cadastro e o Open Finance, justamente os fluxos onde a confiança visual conta mais.
- **Recomendação:** adicionar ao topo de cada `<style>` o bloco já presente nas páginas admin:
  ```css
  @media (hover: none) and (pointer: coarse) {
    *:hover { transform: none !important; box-shadow: none !important; }
  }
  ```
  Idealmente, mover esse bloco para `_foundation/reset.css` para herdar em todas as páginas.

**[ALTO] MB-02 · Topbar admin do `23_basta-diagnostico-admin.html` sem `safe-area-inset`**
- **O quê:** todas as outras páginas com topbar fixa aplicam `padding-top: max(14px, env(safe-area-inset-top))`. A 23 não tem.
- **Impacto:** no iPhone com notch, a logo ou avatar do topbar ficam embaixo da câmera/notch.
- **Recomendação:** adicionar a regra `@supports (padding: env(safe-area-inset-top))` à página 23.

**[MÉDIO] MB-03 · Páginas 03–11 sem `@media (min-width:)`**
- **O quê:** páginas de login/recuperar/Open Finance não têm breakpoints — são puramente single-column mobile-first.
- **Impacto:** em desktops largos, formulários ficam ocupando uma faixa estreita centralizada (intencional pra essas peças). Não é bug, mas vale validar visualmente que o resultado em 1440px+ está como você quer.
- **Recomendação:** abrir essas 9 páginas no Chrome DevTools em 1920×1080 e validar se o `max-width` do container dá uma sensação de "página", não de "form perdido".

**[BAIXO] MB-04 · 22_admin-usuarios e 16_perfil sem `autocomplete` em campo email**
- Ver achado SE-03 abaixo.

## 3. Integração & navegação

### ✓ Pontos fortes

- **Todos os href `*.html` apontam para arquivos existentes** — zero links quebrados.
- **24_basta-diagnostico-cliente-v2** propositalmente sem inbound links (acesso por URL única) — coerente com o uso pretendido.
- Navegação cliente ↔ cliente e admin ↔ admin é consistente: cada página linka para as outras 11 (admin) ou 6 (cliente).

### ⚠ Achados

**[ALTO] IN-01 · `23_basta-diagnostico-admin.html` linka para si mesmo na sidebar com `nav-item--active` mas o `href` é o mesmo arquivo**
- **O quê:** linha 1058 do 23 → `<a href="23_basta-diagnostico-admin.html" class="nav-item nav-item--active">` — clicar no item "Diagnóstico" enquanto está nele mesmo recarrega a página.
- **Impacto:** baixo (não quebra nada), mas é lixo no roteamento — em produção, esse `href` deve ser uma URL canônica do hub de diagnósticos (ex. `/admin/diagnosticos`).
- **Recomendação:** quando virar app, esse link pisca como "estou aqui" sem `href` recarregando — usar `aria-current="page"` + `<button>` ou link sem ação.

**[MÉDIO] IN-02 · Páginas 05, 06, 08, 10, 11 são "destinos órfãos"**
- **O quê:** zero arquivos linkam para essas 5 páginas. São destinos de fluxos via JS (after-cadastro, after-OF, erro-OF) ou por URL externa (link de email).
- **Impacto:** nenhum em produção (são intencionais), mas em revisão visual fica difícil chegar nelas. A pasta provavelmente precisa de um índice-mestre.
- **Recomendação:** criar um `_audit/INDEX.html` ou `00_index.html` com links pra todas as páginas para QA mais rápido.

**[MÉDIO] IN-03 · Diagnóstico cliente atual: arquivo `24_basta-diagnostico-cliente.html` original ainda existe junto com `24_basta-diagnostico-cliente-v2.html`**
- **O quê:** o conteúdo atualizado está em `v2` mas a versão antiga não foi sobrescrita nem renomeada.
- **Impacto:** confusão sobre qual é a versão canônica.
- **Recomendação:** se a v2 for aprovada, renomear `24_basta-diagnostico-cliente.html` → `24_basta-diagnostico-cliente.html.bak` e renomear `v2 → 24_basta-diagnostico-cliente.html`. Ou consolidar via `git mv`.

## 4. Segurança

### ✓ Pontos fortes

- 9/10 inputs `password` em `03_login.html` têm `autocomplete` correto.
- CPF sempre com `inputmode="numeric"` (não vaza para teclado QWERTY).
- Sem CSP atualmente (esperado para snapshot pré-produção), mas estrutura preparada.

### ⚠ Achados

**[CRÍTICO] SE-01 · `02_cadastro.html` e `05_redefinir-senha.html` com `<input type="password">` sem `autocomplete`**

| Arquivo | # passwords | autocomplete configurado | Faltam |
|---|---|---|---|
| 02_basta-cadastro.html | 2 | 0 | 2 |
| 05_basta-redefinir-senha.html | 2 | 0 | 2 |

- **O quê:** sem `autocomplete="new-password"` (no cadastro) e `autocomplete="new-password"` (no redefinir), gerenciadores de senha (1Password, iCloud Keychain, Chrome) não detectam corretamente o fluxo, podendo: (a) não oferecer sugestão de senha forte, (b) auto-preencher com a senha errada, (c) salvar a senha errada após sucesso.
- **Impacto:** segurança de senhas dos usuários no momento mais sensível (criação/troca).
- **Recomendação:** adicionar `autocomplete="new-password"` em todos os inputs `password` nessas páginas. No login (03) já está OK com `current-password`.

**[ALTO] SE-02 · `01_basta-landing.html` com 1 link `target="_blank"` sem `rel="noopener"`**
- **O quê:** `target="_blank"` sem `rel="noopener noreferrer"` permite que a página de destino acesse `window.opener` e potencialmente faça tabnabbing (redirecionar a aba original).
- **Impacto:** vetor de phishing se o link for para um domínio comprometido.
- **Recomendação:** localizar o link e adicionar `rel="noopener noreferrer"`.

```bash
grep -n 'target="_blank"' 01_basta-landing.html
```

**[ALTO] SE-03 · 3 inputs `email` sem `autocomplete`**

| Arquivo | Local |
|---|---|
| 01_basta-landing.html | input email |
| 16_basta-perfil.html | edição de email |
| 22_basta-admin-usuarios.html | criação de admin |

- **Recomendação:** adicionar `autocomplete="email"` (ou `autocomplete="username"` em fluxos de login).

**[MÉDIO] SE-04 · Debug `alert()` em 3 páginas**

| Arquivo | # alert() | Contexto |
|---|---|---|
| 19_basta-admin-conversas.html | 2 | mock de envio mensagem |
| 23_basta-diagnostico-admin.html | 8 | mocks de banner admin (editar, copiar link, etc.) |
| 24_basta-diagnostico-cliente-v2.html | 1 + 1 console.log | mock de assinatura |

- **Impacto:** baixo em produção (são placeholders), mas precisam virar integrações reais antes de ir ao ar.
- **Recomendação:** marcar com `// TODO` e listar em `_audit/INTEGRATION_TODOS.md` antes de release.

**[BAIXO] SE-05 · Scripts inline volumosos**
- 19_admin-conversas: 273 linhas de JS inline
- 16_perfil: 210 linhas
- 01_landing: 184 linhas
- Outras: 100–170 linhas

- **Impacto:** sem CSP atual, sem problema imediato. Quando aplicar `Content-Security-Policy`, scripts inline precisarão de hash ou nonce.
- **Recomendação:** ao implementar CSP, considerar mover JS para `_scripts/` com `defer`.

**[BAIXO] SE-06 · `02_cadastro.html` 9 inputs textuais com possível gap de 2 sem `<label for="..."` ou `aria-label`**
- **Impacto:** acessibilidade (screen readers) e mobile (campo sem associação clara).
- **Recomendação:** validar manualmente os 9 inputs vs. labels associadas e completar.

**[BAIXO] SE-07 · `01_basta-landing.html` com 1 `<img>` sem `alt`**
- **Impacto:** acessibilidade · imagem decorativa precisa de `alt=""` (vazio); imagem informativa precisa de descrição.
- **Recomendação:** identificar e adicionar `alt`.
